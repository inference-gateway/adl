# AGENTS.md — AI Agent Guide for ADL

This file helps AI agents understand how to work effectively with the ADL (Agent Definition Language) repository. It augments `CLAUDE.md` with additional context, workflow patterns, and conventions.

---

## Project Overview

ADL (Agent Definition Language) is a **declarative, vendor-neutral specification for defining AI agents** — their metadata, capabilities, skills, AI provider, services, and runtime configuration. Think of it as "OpenAPI for AI Agents."

**This repository is NOT an application.** It is the **source of truth for the ADL JSON Schema**. The only shipped artifact is `schema/v1/schema.json` (JSON Schema Draft-07). Downstream tools (e.g., [`adl-cli`](https://github.com/inference-gateway/adl-cli)) pin to tagged versions of this schema.

## Architecture

### Schema-first, no runtime code

The entire repository revolves around a single file: `schema/v1/schema.json`. There is no application code, no tests in a traditional sense, and no `package.json` (the ajv CLI tools are installed ad-hoc).

### Schema versioning model

```
adl.inference-gateway.com/v1  ←→  schema/v1/schema.json
adl.inference-gateway.com/v2  ←→  schema/v2/schema.json  (future)
```

- **apiVersion string and directory always move together.**
- Within a major version, **only backwards-compatible additions** are allowed (new optional fields, new `definitions`, new enum values where consumers tolerate unknowns).
- Breaking changes require a new directory (`schema/v2/`) and a new apiVersion (`adl.inference-gateway.com/v2`). v1 is NOT removed when v2 ships.

### Schema structure

The schema defines the following top-level manifest structure:

```
apiVersion: adl.inference-gateway.com/v1  (const string)
kind: Agent                                 (const string)
metadata:
  name        (lowercase + hyphens)
  description (free text)
  version     (semver)
spec:
  capabilities:  { streaming, pushNotifications, stateTransitionHistory }
  card:          { protocolVersion, url, preferredTransport, input/outputModes, docs/icon URLs }
  agent:         { provider (enum), model, systemPrompt, maxTokens, temperature }
  config:        arbitrary key/value maps
  services:      named service definitions (type, interface, factory, description)
  acronyms:      string list
  skills:        [{ id, name, description, tags, schema, examples, input/outputModes, implementation, inject }]
  server:        { port, scheme, debug, auth }
  language:      { go, typescript, rust } (at least one required)
  artifacts:     { enabled }
  hooks:         { post: [commands] }
  scm:           { provider, url, github_app, issue_templates }
  development:   { sandbox: { flox, devcontainer, dockerCompose }, ai: { enabled } }
  deployment:    { type, cloudrun }
```

## Development Environment Setup

### Option A: flox (recommended)

[flox](https://flox.dev) provides a reproducible environment. It provisions Node.js, go-task, and Claude Code, and runs `npm install` for ajv automatically.

```sh
# Install flox first: https://flox.dev/docs/install-flox/
flox activate
```

After activation, `task`, `node`, and `node_modules/.bin/ajv` are on `PATH`.

### Option B: manual

Requires Node.js 24.x (matching CI) and Git:

```sh
git clone https://github.com/inference-gateway/adl
cd adl
npm install --no-save ajv@8 ajv-cli@5 ajv-formats@3
```

## Key Commands

### Schema compilation (must pass before any PR)
```sh
task compile
# or without go-task:
npx ajv compile --spec=draft7 -c ajv-formats -s schema/v1/schema.json
```

### Manifest validation
```sh
task validate -- path/to/manifest.yaml
# or without go-task:
npx ajv validate --spec=draft7 -c ajv-formats -s schema/v1/schema.json -d path/to/manifest.yaml
```

### List available tasks
```sh
task --list
```

## Testing

### Schema validation (CI-equivalent check)

The primary "test" is that the schema compiles successfully under ajv with Draft-07 semantics. This is what CI runs in `validate-schema.yml`:

```sh
task compile
```

### Manifest validation

To verify that real-world manifests are valid against the schema:

```sh
task validate -- path/to/manifest.yaml
```

### Downstream integration testing (optional)

If changing the schema, test the downstream effect on `adl-cli` by pointing its `Taskfile.yml` to your local copy of `schema/v1/schema.json` and regenerating a sample project.

### Common schema validation failures

1. **ajv compilation errors** — typically caused by invalid JSON Schema syntax (e.g., missing `$ref` targets, incorrect `additionalProperties`, invalid regex patterns).
2. **Breaking changes detected** — removing fields, tightening constraints, or making optional fields required within an existing major version.
3. **Format validation failures** — if using format-specific constraints, ensure `ajv-formats` is loaded (`-c ajv-formats`).

## CI/CD Pipeline

### Validate Schema (automatic)
- **Triggers:** Push or PR to `main`
- **Runner:** ubuntu-24.04
- **Steps:** Checkout → Setup Node.js 24.x → `npm install --no-save ajv@8 ajv-cli@5 ajv-formats@3` → `npx ajv compile --spec=draft7 -c ajv-formats -s schema/v1/schema.json`
- **Must pass before merge.**

### Release (manual)
- **Trigger:** `workflow_dispatch` from GitHub Actions tab
- **Runner:** ubuntu-24.04
- **Uses:** semantic-release v25.0.2 with `conventional-changelog-conventionalcommits`
- **What it does:** Parses Conventional Commits since last tag → bumps version → updates `CHANGELOG.md` → creates GitHub release → creates git tag
- **Branches:** `main` → `latest` channel; `rc/*` → `rc` channel (for pre-releases)
- **Tags are immutable once published.**

## Project Conventions

### Commit message convention (Conventional Commits)

Format: `<type>(<scope>): <description>`

| Type | Release bump | When to use |
|---|---|---|
| `feat` | minor | New schema field or capability |
| `fix` | patch | Schema bug fix |
| `impr` | patch | Improvement (not a feature) |
| `refactor` | patch | Refactoring |
| `perf` | patch | Performance |
| `docs` | patch | Documentation |
| `style` | patch | Formatting |
| `test` | patch | Tests |
| `build` | patch | Build/tooling |
| `ci` | patch | CI config |
| `chore` | patch | Miscellaneous (except `chore(release):`) |

**PR titles become squash-merge commits**, so the title must follow this convention.

### Schema editing rules

1. **Strictly additive within a major version.** Only backwards-compatible changes to `schema/v1/`. New optional fields, new `definitions`, new enum values — yes. Removing/renaming/tightening — no.
2. **Breaking changes require a new major version** (`schema/v2/` + `adl.inference-gateway.com/v2`). Discuss via GitHub Discussion/issue first.
3. **Tags are immutable.** Once `vX.Y.Z` is cut, the schema at that tag never changes.
4. **README must be updated** when introducing new top-level fields or capabilities.

### Code style

- JSON Schema follows Draft-07 specification
- Use `$ref` for reusable definitions (see existing pattern in `schema.json`)
- Follow alphabetical ordering of properties within objects (for consistency)
- Use `description` fields to document each property's purpose
- Enum values should be lowercase

## Typical AI Agent Workflows

### 1. Adding a new optional field to the schema

1. Read `schema/v1/schema.json` to understand the existing structure.
2. Read `CLAUDE.md` and understand the additive contract rules.
3. Add the new property (with `type`, `description`, and optionally `default`) to the relevant definition.
4. Run `task compile` to verify.
5. Update `README.md` if the field is a new top-level capability.
6. Use a Conventional Commit title like `feat(schema): Add optional spec.newField`.

### 2. Adding a new enum value (e.g., a new provider)

1. Find the relevant enum array in the schema (e.g., `provider` in the `Agent` definition).
2. Add the new value to the `enum` array.
3. Run `task compile` to verify.
4. Use a commit like `feat(schema): Add provider X to agent providers`.

### 3. Validating a user's manifest

1. Run `task validate -- path/to/manifest.yaml`.
2. If validation fails, parse the ajv error output to identify the schema violation.

### 4. Checking current release state

1. Read `CHANGELOG.md` for the latest version.
2. Check `git tag --list` for all published tags.
3. Check `git log --oneline -n 10` for recent commits.

## Git Branch Strategy

- **`main`** — default branch, protected. All PRs merge here.
- **`rc/*`** — release candidate branches (e.g., `rc/v1.1.0`). Trigger the same `Release` workflow manually from these branches.
- **Feature branches** — named arbitrarily (e.g., `feat/add-mcp-support`). CI runs on PRs to `main`.
